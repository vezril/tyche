# ADR-007: Field-level encryption for Plaid secrets; master key in `.env`, outside the DB and backups

**Status:** Proposed (for Gate 2) · **Date:** 2026-06-12 · **Drivers:** NFR-3 (secrets at rest), NFR-2 (residency), FR-35 (restore semantics), NFR-5 (zero-extra-infra)

## Context

The system holds three durable secrets: the Plaid client secret, one `access_token` per Item (grants ongoing read access to Calvin's bank — "treat like a password," per the feasibility report), and the user's password (hash, not reversible — separate concern, argon2id, NFR-10). NFR-3: no plaintext token in DB, files, or logs. The threat model is honest and small: a stolen backup artifact, a copied database file, or DB contents in logs/dumps — *not* a compromised live host (an attacker with root on the box owns the process memory and any key it can read; no self-hosted design defeats that). FR-35 requires a restore on a new host to resume syncing "after at most a re-link."

## Decision

- Encrypt Plaid `access_token`s and the Plaid client secret with **AES-256-GCM** (random nonce per value, key id in the ciphertext envelope for future rotation) at the persistence boundary; SQLite stores ciphertext only.
- The 256-bit **master key lives in the compose `.env` file** (`MASTER_KEY=`), generated by the first-run setup and never written to the database, application logs, or the backup artifact. Backups (FR-35) contain ciphertext; `.env` is the user's to safeguard separately (one line in the README's backup section).
- **Log hygiene:** the logger applies a redaction layer (deny-list of secret-bearing fields + Plaid token patterns) so tokens cannot appear in logs even on error paths (NFR-3's 30-day log grep test).
- Sync cursors and item ids are not secrets and stay plaintext (they're useless without the token).

## Consequences

**Positive:** a leaked backup or stolen DB file exposes no bank credentials; NFR-3's verification (grep dump/volume/logs) passes by construction; no new infrastructure (meets NFR-5) and no external dependency (meets NFR-2); key rotation is possible (re-encrypt rows, bump key id) without re-linking.

**Negative:** **lose the `.env`, lose the connections** — restore still recovers every transaction and budget cent, but each Item needs a re-link via update mode; this is exactly the FR-35 "at most a re-link" allowance, and Calvin must accept it explicitly (gate item). The key sits in plaintext on the host's disk — acceptable: it moves the secret boundary off the database/backup files, which is what NFR-3 asks; defending against host root is out of scope (AS-2 trusted network, single-user box). Encryption-at-the-app-layer means ad-hoc SQL against the items table shows ciphertext — intended, mildly inconvenient for debugging.

## Alternatives considered

- **Full-database encryption (SQLCipher):** encrypts everything including budget data, but complicates every tool touching the DB (backups, `sqlite3` CLI inspection, the 2035 data-longevity story), needs a maintained native build, and still needs the same key-on-host. NFR-3 only requires secrets encrypted, not the ledger. Rejected as over-broad.
- **External KMS / Vault / cloud key service:** real key custody, but violates NFR-2 (data/keys leaving the host) or adds a Vault container (NFR-5/6) for a one-key problem. Rejected.
- **Docker secrets / OS keyring:** Docker swarm secrets don't fit plain compose cleanly; host keyrings vary by distro — `.env` is the boring, documented, compose-native spot. Rejected for portability.
- **Storing the key in the DB (or no encryption):** fails NFR-3 outright. Rejected.
